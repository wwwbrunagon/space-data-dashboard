🚀 Space Data Dashboard

A senior-level walkthrough building a full-stack dashboard that displays NASA's Astronomy Photo of the Day and live asteroid data — with a PostgreSQL historical archive and cron-based background sync.

What you'll build

By the end of this guide you'll have a production-grade Next.js 15 app that:

  • Fetches the NASA APOD (Astronomy Photo of the Day) and renders it with metadata
  • Displays near-Earth asteroid data from the NeoWs API in a filterable table
  • Stores every daily fetch in PostgreSQL — giving you a queryable historical archive
  • Runs a background cron job to populate history even when no user is online
  • Uses smart caching to avoid hammering the NASA API (1,000 req/day limit)

Prerequisite knowledge

This guide assumes you're comfortable with:

  • TypeScript fundamentals (generics, utility types, discriminated unions)
  • React patterns (Server Components vs Client Components, hooks)
  • SQL basics and understanding of ORMs
  • HTTP, REST, and async/await patterns

Mentor note: I'll explain why we make every architectural decision, not just what to type. At the mid-senior level, understanding the tradeoffs is more valuable than memorising syntax.

Key tech decisions

Why Next.js App Router?

The App Router's Server Components let us fetch data on the server before sending HTML — zero loading spinners for initial data. Route Handlers replace a separate Express server. For a dashboard like this, it collapses a 3-service architecture (frontend + BFF + API) into one coherent codebase.

Why Prisma over raw SQL?

At this scale (2–3 tables), raw SQL is perfectly fine. But Prisma gives us type-safe query results — the TypeScript types for query results are auto-generated from the schema. That's zero type-drift between the DB and your application code. It also handles migrations cleanly.

Why PostgreSQL over SQLite?

SQLite is excellent for local dev. We're choosing Postgres because: (1) JSON column support for flexible NASA response caching, (2) proper full-text search on APOD explanations if you ever want it, and (3) Vercel Postgres / Supabase / Neon all speak Postgres — deployment is trivial.

System Design

Data flow walkthrough

Let's trace a user request through the system to understand the design decisions at each hop:

User opens dashboard
→
Next.js Server Component renders
→
Check DB for today's APOD
→
Cache hit? Serve from DB
Cache miss?
→
Call NASA API
→
Upsert to DB
→
Return to user

Why check DB first? NASA's free tier allows 1,000 requests/day across all API keys. Serving today's already-fetched APOD from the DB is instant (~2ms vs ~400ms) and doesn't consume API quota. The cron job pre-populates the DB so the first user of the day doesn't take the API hit.

Caching strategy: 3-layer defence

Layer 1 — Next.js fetch cache (edge)

Next.js 15 wraps native fetch with an extended cache. We set revalidate: 3600 on NASA API calls. This caches at the CDN/edge layer — subsequent deploys or function restarts still serve stale data within the TTL.

Layer 2 — PostgreSQL as a "smart cache"

We write every NASA response to the DB. On each request, we first query the DB by date. This is our primary cache — it survives Next.js cache invalidations, Vercel function restarts, and API quota resets.

Layer 3 — React's cache()

Within a single render pass, multiple Server Components might request the same data. React's cache() deduplicates these within a request lifecycle — the DB is only hit once per render.

Database design principles

We have three main tables. The key decisions:

  • apod_snapshots: date is the natural primary key (APOD is 1-per-day). We store the raw JSON as a JSONB column alongside typed columns — so we have structured queries and the full original payload for schema migrations.
  • asteroids: Asteroid IDs are NASA's own string IDs (e.g. "3542519"). We use those as PK, not a serial — avoids unnecessary surrogate keys when a natural key exists.
  • sync_log: A lightweight audit table that records every cron run, its status, and how many records were upserted. This is critical for ops — you can instantly see if your cron job has been silently failing.

API key security

Never expose NASA_API_KEY to the browser. All NASA API calls happen in Server Components or Route Handlers. The key lives in .env.local / environment variables and is never referenced in any use client file. Next.js will throw at build time if you try to reference process.env.NASA_API_KEY in a Client Component — but don't rely on that guard.

  new PrismaClient({

  var prisma: ReturnType<typeof createPrismaClient> | undefined

Database Schema

Prisma schema — prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

/// Stores one APOD entry per day. The `date` field IS the primary key
/// because NASA guarantees one photo per calendar date — no need for
/// a surrogate serial ID here. JSONB raw_data lets us cache the full
/// response so we can add new typed columns later without re-fetching.
model ApodSnapshot {
  date        String   @id                  // "YYYY-MM-DD"
  title       String
  explanation String
  mediaType   String                         // "image" | "video"
  url         String
  hdurl       String?
  copyright   String?
  serviceVersion String
  rawData     Json                            // Full NASA response
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@map("apod_snapshots")
}

/// Near-Earth Object. NASA IDs are stable string identifiers —
/// using them as PK avoids a useless surrogate and makes lookups
/// by NASA ID O(1) with no extra index needed.
model Asteroid {
  nasaId              String   @id            // e.g. "3542519"
  name                String
  absoluteMagnitude   Float
  estimatedDiameterMinKm Float
  estimatedDiameterMaxKm Float
  isPotentiallyHazardous Boolean
  closeApproachDate   String                 // "YYYY-MM-DD"
  relativeVelocityKmH Float
  missDistanceKm      Float
  orbitingBody        String
  rawData             Json
  fetchedAt           DateTime @default(now())

  @@index([closeApproachDate])
  @@index([isPotentiallyHazardous])
  @@map("asteroids")
}

/// Audit log for cron runs. Essential for observability —
/// you want to know when the last successful sync ran
/// without tailing server logs.
model SyncLog {
  id          Int      @id @default(autoincrement())
  syncType    String                         // "apod" | "asteroids"
  status      String                         // "success" | "error"
  recordCount Int?
  errorMessage String?
  durationMs  Int?
  ranAt       DateTime @default(now())

  @@index([syncType, ranAt])
  @@map("sync_log")
}

Schema decisions explained

DecisionWhy
rawData JsonStore the full NASA payload alongside typed columns. When NASA adds a new field, you have the data already — no back-fill needed.
@@index([closeApproachDate])The most common query pattern is "asteroids approaching on date X". Without this index, every query is a full table scan.
date @id on ApodSnapshotNatural key eliminates duplicate entries by design — upserting by date is idempotent.
SyncLog.durationMsLets you detect performance degradation in the cron job over time.

Zod schemas — src/types/nasa.ts

/**
 * Zod schemas serve double duty:
 * 1. Runtime validation of NASA API responses (they can change without notice)
 * 2. TypeScript type inference — no manual interface maintenance
 *
 * The pattern: define the Zod schema, then infer the TypeScript type from it.
 * This is the single-source-of-truth approach to typing external data.
 */
import { z } from 'zod'

export const ApodSchema = z.object({
  copyright:       z.string().optional(),
  date:            z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  explanation:     z.string(),
  hdurl:           z.string().url().optional(),
  media_type:      z.enum(['image', 'video']),
  service_version: z.string(),
  title:           z.string(),
  url:             z.string().url(),
})

export type Apod = z.infer<typeof ApodSchema>

// NeoWs (Near Earth Object Web Service) response is deeply nested
const CloseApproachSchema = z.object({
  close_approach_date:       z.string(),
  relative_velocity:         z.object({
    kilometers_per_hour:       z.string().transform(Number),
  }),
  miss_distance:             z.object({
    kilometers:                z.string().transform(Number),
  }),
  orbiting_body:             z.string(),
})

export const AsteroidSchema = z.object({
  id:                           z.string(),
  name:                         z.string(),
  absolute_magnitude_h:         z.number(),
  estimated_diameter:           z.object({
    kilometers:                 z.object({
      estimated_diameter_min:   z.number(),
      estimated_diameter_max:   z.number(),
    }),
  }),
  is_potentially_hazardous_asteroid: z.boolean(),
  close_approach_data:          z.array(CloseApproachSchema),
})

export type Asteroid = z.infer<typeof AsteroidSchema>

export const NeoFeedSchema = z.object({
  near_earth_objects: z.record(z.string(), z.array(AsteroidSchema)),
  element_count:      z.number(),
})

Mentor note on .transform(Number): NASA's API returns numeric values like miss_distance.kilometers as strings. Zod's .transform(Number) handles the coercion at parse time and gives you a number in the inferred type. No manual parsing elsewhere in your app.

NASA API client — src/lib/nasa.ts

import { cache } from 'react'
import { ApodSchema, NeoFeedSchema, type Apod, type Asteroid } from '@/types/nasa'
import { format } from 'date-fns'

const BASE = 'https://api.nasa.gov'
const key = () => process.env.NASA_API_KEY

/**
 * A typed fetch wrapper that:
 * - Appends the API key automatically
 * - Uses Next.js extended fetch caching (revalidate every hour)
 * - Throws a meaningful error on non-2xx responses
 * - Returns parsed + validated JSON via Zod
 */
async function nasaFetch<T>(
  path: string,
  schema: { parse: (data: unknown) => T },
  params: Record<string, string> = {},
  options: RequestInit = {},
): Promise<T> {
  const url = new URL(`${BASE}${path}`)
  url.searchParams.set('api_key', key() ?? 'DEMO_KEY')
  Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v))

  const res = await fetch(url.toString(), {
    next: { revalidate: 3600 },  // Next.js cache: revalidate every 1h
    ...options,
  })

  if (!res.ok) {
    throw new Error(
      `NASA API error: ${res.status} ${res.statusText} on ${path}`
    )
  }

  const json = await res.json()
  return schema.parse(json) // Zod validates + transforms
}

/**
 * React cache() deduplicates calls within a single render pass.
 * If ApodCard and the page title both call fetchApod("2025-04-14"),
 * the DB/API is only hit once. Zero extra config — just wrap with cache().
 */
export const fetchApod = cache(
  async (date: string = format(new Date(), 'yyyy-MM-dd')): Promise<Apod> => {
    return nasaFetch('/planetary/apod', ApodSchema, { date })
  }
)

export const fetchAsteroids = cache(
  async (
    startDate: string,
    endDate: string
  ): Promise<Asteroid[]> => {
    const feed = await nasaFetch(
      '/neo/rest/v1/feed',
      NeoFeedSchema,
      { start_date: startDate, end_date: endDate }
    )
    // Flatten the date-keyed object into a flat array
    return Object.values(feed.near_earth_objects).flat()
  }
)

API Route Handlers

APOD route — src/app/api/apod/route.ts

import { NextRequest, NextResponse } from 'next/server'
import { db } from '@/lib/db'
import { fetchApod } from '@/lib/nasa'
import { format } from 'date-fns'

/**
 * GET /api/apod?date=YYYY-MM-DD
 *
 * Cache-aside pattern:
 * 1. Check DB first (fast, no API quota)
 * 2. If missing, fetch from NASA, persist, then return
 *
 * This endpoint is used by the client for historical browsing.
 * The SSR page uses the server-side fetchApod() directly.
 */
export async function GET(req: NextRequest) {
  const date =
    req.nextUrl.searchParams.get('date') ??
    format(new Date(), 'yyyy-MM-dd')

  // Validate date format before hitting DB or API
  if (!/^\d{4}-\d{2}-\d{2}$/.test(date)) {
    return NextResponse.json(
      { error: 'Invalid date format. Use YYYY-MM-DD' },
      { status: 400 }
    )
  }

  try {
    // Layer 1: Check the DB cache
    const cached = await db.apodSnapshot.findUnique({
      where: { date },
    })

    if (cached) {
      return NextResponse.json(cached, {
        headers: { 'X-Cache': 'HIT' },
      })
    }

    // Layer 2: Fetch from NASA, persist to DB
    const apod = await fetchApod(date)

    const snapshot = await db.apodSnapshot.upsert({
      where:  { date },
      create: {
        date,
        title:          apod.title,
        explanation:    apod.explanation,
        mediaType:      apod.media_type,
        url:            apod.url,
        hdurl:          apod.hdurl,
        copyright:      apod.copyright,
        serviceVersion: apod.service_version,
        rawData:        apod as unknown as Record<string, unknown>,
      },
      update: {},  // No-op on conflict — APOD for a date never changes
    })

    return NextResponse.json(snapshot, {
      headers: { 'X-Cache': 'MISS' },
    })
  } catch (err) {
    console.error('[APOD API]', err)
    return NextResponse.json(
      { error: 'Failed to fetch APOD data' },
      { status: 502 }
    )
  }
}

Why upsert with empty update: {}? This is an intentional no-op upsert. If the record already exists (race condition between two simultaneous requests), do nothing. If it doesn't exist, create it. The empty update block is not a mistake — it's the idiomatic Prisma pattern for "insert-if-not-exists".

Asteroids route — src/app/api/asteroids/route.ts

import { NextRequest, NextResponse } from 'next/server'
import { db } from '@/lib/db'
import { fetchAsteroids } from '@/lib/nasa'
import { format, addDays } from 'date-fns'

export async function GET(req: NextRequest) {
  const today = format(new Date(), 'yyyy-MM-dd')
  const startDate = req.nextUrl.searchParams.get('start') ?? today
  const endDate   = req.nextUrl.searchParams.get('end')
    ?? format(addDays(new Date(startDate), 7), 'yyyy-MM-dd')

  try {
    // Check if we already have data for this date range in the DB
    const cached = await db.asteroid.findMany({
      where: {
        closeApproachDate: {
          gte: startDate,
          lte: endDate,
        },
      },
      orderBy: { closeApproachDate: 'asc' },
    })

    if (cached.length > 0) {
      return NextResponse.json(
        { asteroids: cached, source: 'db' },
        { headers: { 'X-Cache': 'HIT' } }
      )
    }

    // NASA NeoWs: max 7-day window per request
    const asteroids = await fetchAsteroids(startDate, endDate)

    // Batch upsert all asteroids — more efficient than N individual writes
    await Promise.all(
      asteroids.map((a) =>
        db.asteroid.upsert({
          where:  { nasaId: a.id },
          create: {
            nasaId:                    a.id,
            name:                      a.name,
            absoluteMagnitude:         a.absolute_magnitude_h,
            estimatedDiameterMinKm:    a.estimated_diameter.kilometers.estimated_diameter_min,
            estimatedDiameterMaxKm:    a.estimated_diameter.kilometers.estimated_diameter_max,
            isPotentiallyHazardous:    a.is_potentially_hazardous_asteroid,
            closeApproachDate:         a.close_approach_data[0]?.close_approach_date ?? startDate,
            relativeVelocityKmH:       a.close_approach_data[0]?.relative_velocity.kilometers_per_hour ?? 0,
            missDistanceKm:            a.close_approach_data[0]?.miss_distance.kilometers ?? 0,
            orbitingBody:              a.close_approach_data[0]?.orbiting_body ?? 'Earth',
            rawData:                   a as unknown as Record<string, unknown>,
          },
          update: { fetchedAt: new Date() },
        })
      )
    )

    return NextResponse.json(
      { asteroids, source: 'api' },
      { headers: { 'X-Cache': 'MISS' } }
    )
  } catch (err) {
    console.error('[Asteroids API]', err)
    return NextResponse.json(
      { error: 'Failed to fetch asteroid data' },
      { status: 502 }
    )
  }
}

Cron Sync Job

The cron job runs daily at midnight UTC, pre-fetching the new APOD and the next 7 days of asteroid data. This means the first user of the day gets instant responses from the DB, not a cold API call.

Cron route — src/app/api/cron/sync/route.ts

import { NextRequest, NextResponse } from 'next/server'
import { db } from '@/lib/db'
import { fetchApod, fetchAsteroids } from '@/lib/nasa'
import { format, addDays } from 'date-fns'

/**
 * POST /api/cron/sync
 *
 * Security: Protected by a bearer token (CRON_SECRET env var).
 * On Vercel, set up cron via vercel.json and the token automatically.
 * Locally: curl -X POST localhost:3000/api/cron/sync \
 *   -H "Authorization: Bearer your-cron-secret"
 *
 * Vercel Cron calls this with the CRON_SECRET as a header — we validate
 * it before doing any work. This prevents arbitrary internet traffic
 * from exhausting your NASA API quota.
 */
export async function POST(req: NextRequest) {
  const authHeader = req.headers.get('authorization')
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  const today     = format(new Date(), 'yyyy-MM-dd')
  const sevenDays = format(addDays(new Date(), 7), 'yyyy-MM-dd')
  const start = Date.now()
  const results: Record<string, unknown> = {}

  // ── Sync APOD ────────────────────────────────
  try {
    const apod = await fetchApod(today)
    await db.apodSnapshot.upsert({
      where:  { date: today },
      create: {
        date:           today,
        title:          apod.title,
        explanation:    apod.explanation,
        mediaType:      apod.media_type,
        url:            apod.url,
        hdurl:          apod.hdurl,
        copyright:      apod.copyright,
        serviceVersion: apod.service_version,
        rawData:        apod as unknown as Record<string, unknown>,
      },
      update: {},
    })
    await db.syncLog.create({
      data: { syncType: 'apod', status: 'success', recordCount: 1,
              durationMs: Date.now() - start },
    })
    results.apod = { status: 'ok', title: apod.title }
  } catch (err) {
    await db.syncLog.create({
      data: { syncType: 'apod', status: 'error',
              errorMessage: String(err), durationMs: Date.now() - start },
    })
    results.apod = { status: 'error', error: String(err) }
  }

  // ── Sync asteroids ────────────────────────────
  const astStart = Date.now()
  try {
    const asteroids = await fetchAsteroids(today, sevenDays)
    await Promise.all(
      asteroids.map((a) =>
        db.asteroid.upsert({
          where:  { nasaId: a.id },
          create: {
            nasaId:                 a.id,
            name:                   a.name,
            absoluteMagnitude:      a.absolute_magnitude_h,
            estimatedDiameterMinKm: a.estimated_diameter.kilometers.estimated_diameter_min,
            estimatedDiameterMaxKm: a.estimated_diameter.kilometers.estimated_diameter_max,
            isPotentiallyHazardous: a.is_potentially_hazardous_asteroid,
            closeApproachDate:      a.close_approach_data[0]?.close_approach_date ?? today,
            relativeVelocityKmH:    a.close_approach_data[0]?.relative_velocity.kilometers_per_hour ?? 0,
            missDistanceKm:         a.close_approach_data[0]?.miss_distance.kilometers ?? 0,
            orbitingBody:           a.close_approach_data[0]?.orbiting_body ?? 'Earth',
            rawData:                a as unknown as Record<string, unknown>,
          },
          update: { fetchedAt: new Date() },
        })
      )
    )
    await db.syncLog.create({
      data: { syncType: 'asteroids', status: 'success',
              recordCount: asteroids.length, durationMs: Date.now() - astStart },
    })
    results.asteroids = { status: 'ok', count: asteroids.length }
  } catch (err) {
    await db.syncLog.create({
      data: { syncType: 'asteroids', status: 'error',
              errorMessage: String(err), durationMs: Date.now() - astStart },
    })
    results.asteroids = { status: 'error', error: String(err) }
  }

  return NextResponse.json({ synced: results, totalMs: Date.now() - start })
}

{

Cron timing tip: NASA publishes the new APOD at midnight UTC. We schedule our sync at 01:00 UTC to give them a 1-hour buffer — this avoids the race condition where our cron runs before the new photo is live and caches a stale response.

TypeScript Types

Good type architecture means you define types once and let the compiler propagate them everywhere. Here's the full type strategy for this project:

DB query types — src/lib/queries.ts

import { cache } from 'react'
import { db } from './db'
import { type Prisma } from '@prisma/client'

/**
 * Prisma's generated types are deeply nested generics.
 * We use `Prisma.ApodSnapshotGetPayload` to derive the exact
 * TypeScript type that Prisma will return for a given query.
 * This is more precise than importing the raw model type —
 * it accounts for select/include/omit modifiers.
 */
export type ApodSnapshotRow = Prisma.ApodSnapshotGetPayload<{}>

export type AsteroidRow = Prisma.AsteroidGetPayload<{}>

// Typed query functions with React cache() deduplication
export const getApodByDate = cache(
  async (date: string): Promise<ApodSnapshotRow | null> => {
    return db.apodSnapshot.findUnique({ where: { date } })
  }
)

export const getAsteroidsByDateRange = cache(
  async (startDate: string, endDate: string): Promise<AsteroidRow[]> => {
    return db.asteroid.findMany({
      where: {
        closeApproachDate: { gte: startDate, lte: endDate },
      },
      orderBy: [
        { isPotentiallyHazardous: 'desc' }, // Hazardous first
        { closeApproachDate: 'asc' },
      ],
    })
  }
)

export const getRecentApodHistory = cache(
  async (limit = 30): Promise<Pick<ApodSnapshotRow, 'date' | 'title' | 'url'>[]> => {
    return db.apodSnapshot.findMany({
      orderBy: { date: 'desc' },
      take: limit,
      select: { date: true, title: true, url: true },
    })
  }
)

export const getLastSyncStatus = cache(
  async () => {
    const [apodLog, asteroidLog] = await Promise.all([
      db.syncLog.findFirst({ where: { syncType: 'apod' }, orderBy: { ranAt: 'desc' } }),
      db.syncLog.findFirst({ where: { syncType: 'asteroids' }, orderBy: { ranAt: 'desc' } }),
    ])
    return { apodLog, asteroidLog }
  }
)

Mentor note on Pick<>: The getRecentApodHistory query uses a select in Prisma. Instead of returning ApodSnapshotRow[] (which would be wrong — those rows don't have all fields), we return Pick<ApodSnapshotRow, 'date' | 'title' | 'url'>[]. This is honest typing — it exactly matches what Prisma returns. Never lie to TypeScript with as when you can express the actual shape.

Client Components

Asteroid table — src/components/AsteroidTable.tsx

'use client'
/**
 * Client Component — "use client" is the boundary directive.
 *
 * Why client here? Because the user needs to:
 * - Filter by hazardous/all
 * - Sort columns by clicking headers
 * - Paginate the table
 *
 * All of these require local state (useState) and event handlers.
 * The INITIAL data is passed down from the parent Server Component as
 * props — no client-side fetch on mount. React Query handles
 * subsequent fetches (e.g. when user changes the date range).
 */
import { useState, useMemo } from 'react'
import { type AsteroidRow } from '@/lib/queries'

type SortKey = keyof Pick<
  AsteroidRow,
  'closeApproachDate' | 'missDistanceKm' | 'relativeVelocityKmH' | 'estimatedDiameterMaxKm'
>

interface Props {
  initialAsteroids: AsteroidRow[]
}

export default function AsteroidTable({ initialAsteroids }: Props) {
  const [showHazardousOnly, setShowHazardousOnly] = useState(false)
  const [sortKey, setSortKey] = useState<SortKey>('closeApproachDate')
  const [sortAsc, setSortAsc] = useState(true)
  const [page, setPage] = useState(1)
  const PAGE_SIZE = 10

  const filtered = useMemo(() => {
    const data = showHazardousOnly
      ? initialAsteroids.filter((a) => a.isPotentiallyHazardous)
      : initialAsteroids

    return [...data].sort((a, b) => {
      const aVal = a[sortKey]
      const bVal = b[sortKey]
      const cmp = typeof aVal === 'string'
        ? aVal.localeCompare(bVal as string)
        : (aVal as number) - (bVal as number)
      return sortAsc ? cmp : -cmp
    })
  }, [initialAsteroids, showHazardousOnly, sortKey, sortAsc])

  const totalPages = Math.ceil(filtered.length / PAGE_SIZE)
  const paginated = filtered.slice((page - 1) * PAGE_SIZE, page * PAGE_SIZE)

  function toggleSort(key: SortKey) {
    if (sortKey === key) {
      setSortAsc((prev) => !prev)
    } else {
      setSortKey(key)
      setSortAsc(true)
    }
    setPage(1)
  }

  return (
    <div className="space-y-4">
      <div className="flex items-center gap-4">
        <label className="flex items-center gap-2 text-sm text-white/70 cursor-pointer">
          <input
            type="checkbox"
            checked={showHazardousOnly}
            onChange={(e) => { setShowHazardousOnly(e.target.checked); setPage(1) }}
            className="accent-red-500"
          />
          Show potentially hazardous only
        </label>
        <span className="text-sm text-white/40">
          {filtered.length} asteroids
        </span>
      </div>

      <div className="overflow-x-auto rounded-lg border border-white/10">
        <table className="w-full text-sm">
          <thead>
            <tr className="border-b border-white/10 bg-white/5">
              <th className="text-left p-3 text-white/60 font-medium">Name</th>
              <th className="text-left p-3 text-white/60 font-medium">Hazardous</th>
              {([
                ['closeApproachDate', 'Date'],
                ['missDistanceKm', 'Miss dist. (km)'],
                ['relativeVelocityKmH', 'Velocity (km/h)'],
                ['estimatedDiameterMaxKm', 'Max Ø (km)'],
              ] as const).map(([key, label]) => (
                <th
                  key={key}
                  className="text-left p-3 text-white/60 font-medium cursor-pointer hover:text-white select-none"
                  onClick={() => toggleSort(key)}
                >
                  {label} {sortKey === key ? (sortAsc ? '↑' : '↓') : '↕'}
                </th>
              ))}
            </tr>
          </thead>
          <tbody>
            {paginated.map((a) => (
              <tr
                key={a.nasaId}
                className="border-b border-white/5 hover:bg-white/5 transition-colors"
              >
                <td className="p-3 text-white font-medium">{a.name}</td>
                <td className="p-3">
                  {a.isPotentiallyHazardous ? (
                    <span className="text-red-400 font-semibold">⚠ YES</span>
                  ) : (
                    <span className="text-green-500">✓ No</span>
                  )}
                </td>
                <td className="p-3 text-white/70">{a.closeApproachDate}</td>
                <td className="p-3 text-white/70">
                  {a.missDistanceKm.toLocaleString('en-US', { maximumFractionDigits: 0 })}
                </td>
                <td className="p-3 text-white/70">
                  {a.relativeVelocityKmH.toLocaleString('en-US', { maximumFractionDigits: 0 })}
                </td>
                <td className="p-3 text-white/70">
                  {a.estimatedDiameterMaxKm.toFixed(3)}
                </td>
              </tr>
            ))}
          </tbody>
        </table>
      </div>

      <div className="flex items-center justify-between text-sm text-white/50">
        <span>Page {page} of {totalPages}</span>
        <div className="flex gap-2">
          <button
            onClick={() => setPage((p) => Math.max(1, p - 1))}
            disabled={page === 1}
            className="px-3 py-1 rounded border border-white/20 disabled:opacity-30 hover:bg-white/10"
          >← Prev</button>
          <button
            onClick={() => setPage((p) => Math.min(totalPages, p + 1))}
            disabled={page === totalPages}
            className="px-3 py-1 rounded border border-white/20 disabled:opacity-30 hover:bg-white/10"
          >Next →</button>
        </div>
      </div>
    </div>
  )
}

/**

 *
 * Architecture pattern: "Fetch at the top, pass down as props."
 *
 * All data fetching happens here (or in child Server Components).

NASA API rate limit handling

// In src/lib/nasa.ts — enhanced nasaFetch with retry logic
async function nasaFetch<T>(
  path: string,
  schema: { parse: (data: unknown) => T },
  params: Record<string, string> = {},
  retries = 2,
): Promise<T> {
  const url = new URL(`${BASE}${path}`)
  url.searchParams.set('api_key', process.env.NASA_API_KEY ?? 'DEMO_KEY')
  Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v))

  for (let attempt = 0; attempt <= retries; attempt++) {
    const res = await fetch(url.toString(), {
      next: { revalidate: 3600 },
    })

    if (res.status === 429) {
      // Rate limited — respect the Retry-After header
      const retryAfter = Number(res.headers.get('Retry-After') ?? 60)
      if (attempt < retries) {
        await new Promise((r) => setTimeout(r, retryAfter * 1000))
        continue
      }
      throw new Error(`NASA API rate limited. Retry after ${retryAfter}s`)
    }

    if (!res.ok) {
      throw new Error(`NASA API ${res.status}: ${res.statusText}`)
    }

    const json = await res.json()
    return schema.parse(json)
  }

  throw new Error('NASA API fetch failed after retries')
}

Production observability tip: Add the error's digest (a hash Next.js generates for server-side errors) to your error page UI. When a user reports an issue, they can share the digest and you can correlate it with server logs instantly.
